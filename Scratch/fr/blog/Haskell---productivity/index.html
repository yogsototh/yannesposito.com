Haskell & productivity

tlpl: Savoir quand choisir Haskell.

Many people have only heard about Haskell and don’t know what the fuss is all about. Some might not even heard about this language.

If I were one of them, I would simply, say:

“Yet another programming language. Just a new syntax to learn.”

How wrong would I’ve been. But, one thing about this is true. Haskell is also a tool than can be used. And as all tools, there are time when this is the right tool for the job. And there are time when not.

So, as always, I start by the conclusion:

Haskell not best for Haskell is best for
Prototyping Bug free prototyping
Show something dirty Show something working without corner case
very small project Medium to big project and to very big projects

To promote Haskell, I’ll compare it to some other language. Particularly a dynamic typed language. Let’s choose javascript for a first example.

I’ll use the actionhero.js minimal node framework.

It is a very good node framework to create scalable JSON API. Most good choices were made for you. It helps to make node feel a bit less “low level”.

Here is the code for a route that will sum two parameters:

> curl 'http://localhost:8080/api/addition?val1=1&val2=2'

Should return 3 in a JSON object. This is not RESTful at all, but we don’t care for the purpose of this article.

Here is the js code of the addition action:

var action = {};

action.name          = 'addition';
action.description   = 'I add things';
action.inputs        = {'required': ['val1','val2'], 'optional': [] }
action.outputExample = { result: 3 }
action.blockedConnectionTypes = [];

action.run = function(api,connection, next) {
    // We add here
    connection.response.sum = connection.params.val1 +
                              connection.params.val2;
    next(connection,true);
}

exports.action = action;

So everything seems right to me. Now let’s give it a try:

Yes, you read it well. The answer was "12" not 3. curl send parameters as String. And "1" + "2" is "12" in javascript. Bad, we’ll have to check the type of our parameters. Each one of them, if we want them to work in all cases.

That was a good chance to be able to detect it. Even with basic unit testing this error might have passed unnoticed.

To protect ourselves from our own dumb, or I should say lack of full knowledge of the programming environment we use, we invented many workarounds.

  1. Testing environment – use an environment before releasing to production
  2. Unit testing – automatize a lot of tests even before launching to testing environment
  3. REPL – Try the behaviour of a function before testing it
  4. Type checking – verify the type of objects before trying to use them

Coming from C, C++ and Java, you soon realize that type checking is a burden more than a helper. You pass your time to converting data between different equivalent types.

The solution of dynamic typed languages is to in fact try to convert data for you by trying to guess which type you really wanted. The result is an immediate cleaner syntax.

The problem with the full dynamic typed languages is that they throw away all advantages of static typing. In fact, a better word for dynamic typed languages would be unityped languages.

So, is there a solution that could bring all advantages of static typing and automatic type conversion?

This is what Haskell brings to the table.

You program with a clean syntax but continue to have type verification made for you.
